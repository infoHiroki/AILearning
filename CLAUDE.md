## 🎵 TTS（音声変換）ルール・ガイドライン

### 📝 TTS台本作成の基本原則

#### 🎯 台本準備ルール
1. **マークダウン記号は台本で直接削除**
   - プログラムでクリーンアップせず、台本ファイルで修正
   - 技術的負債を避ける（KISS原則）

2. **見出し部分の句読点追加**
   - 元見出し（#, ##, ###）だった部分に句点「。」を追加
   - 800ms程度の自然なポーズを作る
   - 例：`はじめに` → `はじめに。`

3. **列挙・項目説明での間改善**
   - 各項目説明後に読点「、」を追加
   - 例：`1つ目は役割。` → `1つ目は役割、`
   - 聞き手の理解を助ける適切なポーズを作成

4. **例文・引用での間改善**
   - 導入部分に読点を追加
   - 例：`良い例はこんな感じです。` → `良い例は、こんな感じです。`
   - 引用文前後で自然な区切りを作る

5. **接続詞での間改善**  
   - 「そこで」「また」「最後に」等の後に読点追加
   - 例：`そこで次は` → `そこで、次は`
   - 話の流れをより自然にする

#### 🔧 技術設定

##### 音声品質設定（Google TTS）
```python
voice = texttospeech.VoiceSelectionParams(
    language_code="ja-JP",
    name="ja-JP-Neural2-B",  # Neural2エンジン（最高品質）
    ssml_gender=texttospeech.SsmlVoiceGender.FEMALE
)

audio_config = texttospeech.AudioConfig(
    audio_encoding=texttospeech.AudioEncoding.MP3,
    speaking_rate=0.9,        # やや遅め（自然なポーズ）
    pitch=0.0,               # 標準ピッチ  
    volume_gain_db=0.0,      # 標準音量
    sample_rate_hertz=24000  # 高品質
)
```

##### ファイル管理ルール
1. **命名規則：タイムスタンプ（YYYYMMDD_HHMM）**
   - ❌ 悪い例：`プロンプト基礎_TTS_最終版.mp3`
   - ✅ 良い例：`プロンプト基礎_TTS_20250108_1430.mp3`
   - 「最終版」等の意味のない命名は禁止

2. **フォルダ構成**
   ```
   tts_converter/
   ├── audio_output/          # 生成音声ファイル
   │   ├── [元ファイル名]_YYYYMMDD_HHMM.mp3
   │   └── ...
   ├── tts_converter.py       # 変換プログラム
   └── main.py               # CLI interface
   ```

3. **台本ファイルの配置**
   - 元MDファイルと同階層に`[元ファイル名]_TTS.md`として作成
   - 例：`プロンプト基礎.md` → `プロンプト基礎_TTS.md`

#### 🎪 品質チェック項目

1. **台本チェック**
   - [ ] マークダウン記号（#, ##, *, **, `, ```等）がない
   - [ ] 見出し部分に句点「。」追加済み
   - [ ] 列挙部分の読点「、」調整済み
   - [ ] 接続詞後の読点調整済み

2. **音声品質チェック**
   - [ ] Neural2-B音声でクリアな発音
   - [ ] 適切なポーズ（句点800ms、読点300-400ms）
   - [ ] 自然な話速（speaking_rate: 0.9）
   - [ ] マークダウン記号の読み上げがない

3. **ファイル管理チェック**
   - [ ] タイムスタンプ命名規則に従っている
   - [ ] audio_outputフォルダに整理済み
   - [ ] 不要な重複ファイルを削除済み

### 💡 効率的な作業フロー

1. **台本作成**: 元MDから_TTS.mdを作成し、上記ルールで修正
2. **TTS変換**: `python3 main.py [台本] [出力ファイル名_YYYYMMDD_HHMM.mp3]`
3. **品質確認**: 生成音声を聞いて自然さを確認
4. **必要に応じて台本微調整**: プログラム変更より台本修正を優先

### ⚠️ 重要な注意事項

- **技術的負債回避**: プログラムでの自動クリーンアップより、台本の直接修正を優先
- **KISS原則**: シンプルで保守しやすいソリューションを選択
- **一貫性**: すべてのTTS音声で同じ品質基準を適用
- **タイムスタンプ管理**: バージョン管理は意味のある命名でなくタイムスタンプで実施

#### ⚡ アトミックコミットの原則

   - 破綻した中間状態のコミットは避ける
   - 必要な依存関係も含めてコミット
